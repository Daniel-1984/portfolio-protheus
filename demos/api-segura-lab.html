O que é uma API?

Analogia do dia a dia
Pensa num restaurante. Você (o cliente) não vai até a cozinha buscar sua comida. Você fala com o garçom. O garçom leva seu pedido para a cozinha, a cozinha prepara, e o garçom traz de volta. A API é esse garçom — ela recebe pedidos, processa, e devolve a resposta.
Sem API (acesso direto)

Você vai diretamente ao banco de dados. Qualquer pessoa pode ler, alterar ou apagar tudo. Sem controle, sem segurança, sem registro.

Com API segura

Você se autentica, a API verifica quem você é, valida o que você pediu, registra tudo e só então acessa os dados. Muito mais seguro.

Como funciona tecnicamente

Uma requisição HTTP tem quatro partes principais:

# Exemplo de chamada real (PowerShell)
$headers = @{ "x-api-key" = "MINHA-CHAVE-SEGURA-123" }
$body    = @{ pedido = "PED-10001"; cliente = "TESTE"; valor = 150.75 }

Invoke-RestMethod
  -Uri         "http://127.0.0.1:8000/pedido"   # ENDPOINT — onde
  -Method      Post                              # VERBO — o que fazer
  -Headers     $headers                          # CABEÇALHOS — credenciais
  -Body        $body                             # CORPO — os dados
  -ContentType "application/json"

Autenticação — Quem é você?

Analogia
É o porteiro do prédio com lista de convidados. Sem nome na lista, não entra. Na v1 usamos uma chave fixa (como um cartão de acesso único para todos). Na v2 usamos JWT — cada pessoa tem seu próprio crachá com data de validade.

v1 — Chave fixa (API Key)

  • Uma chave para todos
  • Nunca expira
  • Se vazar, comprometeu tudo
  • Não sabe quem usou

v2 — JWT por usuário

  • Token individual por usuário
  • Expira em 30 minutos
  • Se vazar, só aquele usuário
  • Log com nome do usuário

JWT — O crachá inteligente

JWT significa JSON Web Token. É um texto codificado em 3 partes separadas por ponto. Parece embaralhado, mas tem estrutura.

eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9
.
eyJzdWIiOiJvcGVyYWRvciIsInBlcmZpbCI6Ik9QRVJBRE9SIiwiZXhwIjoxNzQzNTQwMDAwfQ
.
SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c
Parte 1 — Header (azul)

Tipo do token e algoritmo de assinatura.

{ "alg": "HS256", "typ": "JWT" }
Parte 2 — Payload (verde)

Os dados do usuário. Quem é e quando expira.

{
  "sub":    "operador",
  "perfil": "OPERADOR",
  "exp":    1743540000  ← timestamp
}
Parte 3 — Assinatura (roxo)

A API assina o token com uma chave secreta. Se alguém alterar qualquer dado do token, a assinatura fica inválida e a API rejeita. Por isso não dá para falsificar um JWT.

Analogia
O JWT é como um crachá de evento com holograma. Você pode ler o que está escrito (nome, cargo, data de validade), mas não consegue falsificar o holograma porque só a organização tem o carimbo original.

Como o código gera e valida o token

# GERAR token — feito no login
def gerar_token(usuario: str, perfil: str):
    payload = {
        "sub":    usuario,             # quem é
        "perfil": perfil,              # o que pode fazer
        "exp":    agora + 30min,       # quando expira
        "iat":    agora,               # quando foi criado
    }
    return jwt.encode(payload, SECRET_KEY, algorithm="HS256")

# VALIDAR token — em cada requisição protegida
def validar_token(credentials):
    try:
        payload = jwt.decode(token, SECRET_KEY)   # verifica assinatura + validade
        return payload                           # retorna {sub, perfil, exp}
    except jwt.ExpiredSignatureError:
        raise HTTPException(401, "Token expirado")
    except jwt.InvalidTokenError:
        raise HTTPException(401, "Token inválido")

Validação — O que você está mandando?

Analogia
É o formulário de cadastro com campos obrigatórios. Tentou deixar CPF em branco? Colocou letra no campo de número? O sistema rejeita antes de tentar processar. Isso evita dados sujos no banco e ataques de injeção.

Modelo PedidoSC5 — v2 com campos Protheus

class PedidoSC5(BaseModel):
    c5_num:     str   = Field(..., min_length=6, max_length=6)   # exatamente 6 chars
    c5_client:  str   = Field(..., max_length=14)              # código SA1
    c5_lojacli: str   = Field(..., max_length=2)               # 2 dígitos
    c5_emissao: str   = Field(...)                               # YYYYMMDD
    c5_valor:   float = Field(..., gt=0, le=9999999.99)        # > 0

    model_config = { "extra": "forbid" }  # rejeita campos extras

    # Validador customizado — só aceita alfanumérico
    @field_validator("c5_num")
    def valida_num(cls, v):
        if not v.isalnum():
            raise ValueError("c5_num: apenas letras e números")
        return v.upper()
Campo Regra Por quê protege
min_length / max_length Tamanho mínimo e máximo Evita campo vazio ou texto enorme (buffer overflow, spam)
gt=0 / le=999999 Número maior que zero, limite máximo Evita valor negativo ou absurdo (manipulação de pedido)
isalnum() Só letras e números Evita '; DROP TABLE-- (SQL injection)
extra: forbid Rejeita campos não declarados Evita enviar perfil=ADMIN escondido no JSON

Rate Limit — Fila de banco

Analogia
É a senha de atendimento no banco. Você pode retirar no máximo X senhas por hora. Se um robô tentar retirar 1000 senhas em 1 segundo, o sistema bloqueia automaticamente. Isso protege contra ataques de força bruta e sobrecarga.

Diferença entre v1 e v2

v1 — problema: mesmo sem token, as requisições inválidas consumiam o rate limit do IP. Um atacante podia esgotar a janela de um IP legítimo só tentando sem chave.

v2 — solução: rate limit separado por tipo de acesso:

RATE_LIMITS = {
    "autenticado": { "requests": 10, "window": 60 },  # 10 req/min
    "anonimo":     { "requests": 3,  "window": 60 },  # 3 req/min (mais restrito)
}

Como funciona o controle em memória

# Guarda horário de cada requisição por IP
request_store = {
    "127.0.0.1:autenticado": [10:00:01, 10:00:02, 10:00:05],
    "127.0.0.1:anonimo":     [10:00:00],
}

# A cada requisição:
# 1. Remove timestamps mais velhos que 60s
# 2. Se sobrou >= limite → rejeita com 429
# 3. Senão → adiciona timestamp atual e continua
Produção real: usa Redis ao invés de memória, para funcionar em múltiplos servidores. Em lab local, memória é suficiente para aprender o conceito.

Logs — A câmera de segurança

Analogia
É a câmera do prédio. Ela não impede o crime, mas registra tudo. Se algo der errado, você consegue reconstruir o que aconteceu: quem tentou entrar, quando, quantas vezes, com quê.

Códigos HTTP — O que cada número significa

Quando a API responde, ela sempre manda um número. Esse número diz o que aconteceu:

Código Nome O que significa Quando acontece aqui
200 OK Sucesso Funcionou perfeitamente GET /pedido retornou dados
201 Created Criado Recurso criado com sucesso POST /pedido criou SC5
401 Unauthorized Não autorizado Não identificou quem você é Token ausente, expirado ou inválido
403 Forbidden Proibido Te identifiquei, mas você não pode OPERADOR tentou pedido > R$ 50.000
422 Unprocessable Dado inválido O dado enviado não passou na validação Caractere inválido, campo vazio, número negativo
429 Too Many Muitas requisições Passou do limite de chamadas Rate limit excedido
Diferença entre 401 e 403
401 = "Não sei quem você é. Se identifique."
403 = "Sei quem você é. Mas você não tem permissão para isso."
