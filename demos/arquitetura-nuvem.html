Fundamentos

O que é a Nuvem, afinal?

Nuvem (ou Cloud Computing) significa usar computadores, servidores e programas que ficam em grandes data centers espalhados pelo mundo — acessados pela internet, sob demanda.

Analogia do mundo real: Imagine que você vai alugar um escritório num prédio comercial. Você não precisa comprar o prédio, instalar ar-condicionado, contratar segurança ou pagar a conta de luz da estrutura — isso é responsabilidade do dono do prédio. Você só paga pelo espaço que usa, e pode expandir ou diminuir seu andar quando quiser. É assim que a nuvem funciona — o "prédio" pertence a empresas como AWS, Azure ou Google Cloud. Você usa e paga pelo que consumir.

Modelos de Serviço

As 3 camadas da Nuvem

Existem três formas de contratar serviços em nuvem. Cada uma define quanto você controla e quanto o provedor cuida pra você.

IaaS
Infrastructure as a Service

É como alugar um terreno vazio. Você recebe a estrutura bruta — servidores, rede, armazenamento — e constrói tudo por cima do jeito que quiser.

  • AWS EC2 (máquinas virtuais)
  • Azure Virtual Machines
  • Google Compute Engine
  • Você configura o sistema operacional
PaaS
Platform as a Service

É como alugar um apartamento semi-mobiliado. Já tem cozinha, banheiro instalados — você só traz seus móveis (seu código/aplicação).

  • Azure App Service
  • Google App Engine
  • Heroku / Railway
  • Você só gerencia a aplicação
SaaS
Software as a Service

É como ficar num hotel. Tudo pronto, tudo funcionando. Você só usa — sem se preocupar com nada por baixo.

  • Google Workspace (Gmail, Drive)
  • Microsoft 365
  • Salesforce
  • Você só usa o software
Arquitetura

Como as camadas se conectam

Uma arquitetura em nuvem típica funciona como um sistema de camadas. Passe o mouse em cada linha para entender o que ela faz.

Usuário

👤 Interface — Onde o usuário está

Browser, app mobile ou desktop. O ponto de entrada. O usuário nunca vê o que está por baixo.

CDN / DNS

🌍 Roteamento Global

CDN = Rede de Distribuição de Conteúdo. Como um carteiro inteligente que entrega do servidor mais próximo de você, não do mais distante. Resultado: velocidade.

API Gateway

Porteiro da Aplicação — API Gateway

Recebe todas as requisições, verifica quem pode entrar (autenticação), distribui o tráfego e protege contra sobrecarga. É a recepção do prédio.

Aplicação

⚙️ Serviços e Microsserviços

O coração do sistema. Aqui ficam as regras de negócio — cálculos, lógica, integrações. Pode ser uma aplicação monolítica ou vários microsserviços menores conversando entre si.

Dados

Banco de Dados + Armazenamento

Onde tudo é guardado. SQL para dados estruturados (tabelas, registros), NoSQL para dados flexíveis, Blob Storage para arquivos, NFes, PDFs — tudo replicado e com backup automático.

Análise

On-Premise vs. Nuvem

On-premise significa que os servidores ficam fisicamente na sua empresa. Veja como os dois modelos se comparam no dia a dia.

Critério On-Premise (Servidor Local) Cloud (Nuvem)
Investimento inicial Alto — compra de hardware, racks, nobreaks, ar-condicionado Baixo — você paga pelo uso, sem comprar nada físico
Escalabilidade Lenta — precisa comprar e instalar novos servidores Instantânea — mais recursos em minutos via painel
Manutenção Sua equipe cuida de hardware, SO, patches, backups Provedor cuida da infraestrutura; você cuida só da aplicação
Disponibilidade Depende da sua estrutura — um problema de energia pode derrubar tudo SLA de 99,9% a 99,99% — data centers redundantes globalmente
Segurança Você controla totalmente — mas também é responsável por tudo Provedores investem bilhões em segurança (certificações ISO, SOC)
Custo operacional Fixo — paga mesmo sem usar (luz, cooling, equipe) Variável — paga só o que consumir (pay-as-you-go)
Controle total Total — você decide cada detalhe da infraestrutura Parcial — você gerencia a aplicação, o provedor gerencia o resto
Realidade do mercado: A maioria das empresas modernas usa uma arquitetura híbrida — mantendo sistemas legados ou dados sensíveis on-premise, e usando a nuvem para escalabilidade, novas aplicações e integrações. É o modelo mais comum em implantações ERP como o TOTVS Protheus.
