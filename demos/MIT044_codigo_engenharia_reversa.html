03 Funções Detalhadas
F44Pre() VALIDAÇÃO
Porta de entrada da tela. Roda SQL JOIN entre SC6 e SB1 para saber se tem algum produto com B1_UPRCK > 0.
LÓGICA
SQL: SC6 JOIN SB1 WHERE B1_UPRCK > 0 AND C6_NUM = pedido
COUNT = 0 → Help() + Return .F. (bloqueia abertura)
COUNT > 0 → .T. (abre tela)
F44Load() CARGA GRID
Popula o grid da tela de solicitação. Traz APENAS os produtos que podem ser vendidos por KG (B1_UPRCK > 0).
LÓGICA
SQL idêntico ao F44Pre, mas traz C6_ITEM, C6_PRODUTO, valores...
Monta aLoad com XX_CHECK = .F. (desmarcado por padrão)
Retorna array para o MVC popular o grid
F44Mark() UI
Controla os botões "Marcar Todos", "Desmarcar Todos" e "Inverter Seleção" da tela.
LÓGICA
Itera oModelSC6:Length() linhas
lInverter = .T. → nega o valor atual de XX_CHECK
Caso contrário, aplica lMarcar (.T. ou .F.)
Chama oView:Refresh("VIEW_SC6") para atualizar visual
F44Commit() GRAVAÇÃO
Coração da solicitação. Coleta marcados, grava ZPK e dispara e-mail.
LÓGICA
Itera grid, coleta XX_CHECK = .T. → aMarcados
Len(aMarcados) = 0 → Help + Return .F.
Begin Transaction → RecLock ZPK → grava campos
Monta corpo do e-mail texto e chama U_FAT044M()
⚠️ Não verifica se já existe ZPK para o pedido+item. Pode criar duplicata se solicitado 2x.
FAT044M() + F44MEnv() E-MAIL
FAT044M busca e-mails dos usuários do MV_UUSRPKG via PswOrder/PswSeek. F44MEnv usa TMailManager para enviar SMTP.
PARÂMETROS SMTP USADOS
MV_RELSERV → servidor SMTP
MV_RELFROM → remetente
MV_RELAUSU + MV_RELAPSW → credenciais
MV_PORSMTP (padrão 25) + MV_RELAUTH + MV_RELTLS
⚠️ MV_UUSRPKG separado por ";" mas LiberacaoPrecoPorKG usa "$" (substring). Formatos diferentes!
F44LCommit() APROVAÇÃO
Processa a liberação pelo aprovador. Pergunta confirmação, chama F44LPed, atualiza ZPK_APROV.
LÓGICA
MsgYesNo → "Confirma esta operação?"
nPrecoKg > 0 → cAprov = "S" → chama F44LPed()
nPrecoKg = 0 → cAprov = "N" (reprova sem tocar no pedido)
RecLock ZPK → grava ZPK_PRCKG e ZPK_APROV
F44LPed() EXECAUTO
Atualiza o pedido de venda via MSExecAuto(MATA410). Monta arrays de cabeçalho e todos os itens do pedido.
FÓRMULA APLICADA
nPrecoVenda = nPrecoKg × nPesoKg × nQuant
Item liberado → C6_PRCVEN = nPrecoVenda + C6_UPRCKG = "S"
Demais itens → mantém C6_PRCVEN e C6_UPRCKG originais
⚠️ C5_TIPO hardcoded como "N". Se o pedido original for outro tipo, pode gerar inconsistência.
F44CZKC() [M410STTS] AUTO KG
Roda no PE M410STTS ao salvar pedido. Percorre todos os itens e verifica na ZKC se há preço negociado.
LÓGICA
While SC6 → itera itens do pedido
ZKC seek por Filial+Cliente+Loja+Produto
SB1 seek → pega B1_UKILO (peso)
nNovoPreco = ZKC_PRPKG × B1_UKILO × C6_QTDVEN
RecLock SC6 → grava C6_PRCVEN, C6_VALOR, C6_UPRCKG="S"
⚠️ C6_VALOR = nNovoPreco × nQuantidade — duplica a quantidade no cálculo!

05 Bugs & Divergências Código vs Spec
🔴 CRÍTICO M410STTS.PRW
C6_VALOR duplica a quantidade
A função F44CZKC calcula nNovoPreco = ZKC_PRPKG × B1_UKILO × C6_QTDVEN e depois grava C6_VALOR = nNovoPreco × nQuantidade. Isso multiplica C6_QTDVEN duas vezes.
// ERRADO nNovoPreco := nPrecoKgZKC * nPesoKg * nQuantidade SC6->C6_VALOR := nNovoPreco * nQuantidade // ← QTD já está em nNovoPreco!
// CORRETO nNovoPreco := nPrecoKgZKC * nPesoKg * nQuantidade SC6->C6_VALOR := nNovoPreco // ← sem multiplicar de novo
🔴 CRÍTICO tela.tlpp + liberacao.tlpp
Formato do MV_UUSRPKG inconsistente
O arquivo de e-mail separa usuários por ";" e usa StrTokArr(). Já o arquivo de liberação verifica com o operador "$" (substring). Se o parâmetro for "000044;000055", o $ vai fazer "000044" $ "000044;000055" = .T., mas "000044" $ "000055" = .F. — pode funcionar por sorte ou quebrar em cenários específicos.
// liberacao.tlpp If !(cUsuario $ cUsuarios) // "000044" $ "000044;000055" → .T. OK // "0004" $ "000044;..." → .T. FALSO POSITIVO!
// CORRETO: padronizar verificação aUsuarios := StrTokArr(cUsuarios, ";") If AScan(aUsuarios, {|x| AllTrim(x) == cUsuario}) == 0
🟡 AVISO tela.tlpp / F44Commit
Sem verificação de duplicata na ZPK
F44Commit verifica MsSeek antes de gravar, mas só se o pedido+item já existe. Porém, se o aprovador reprova (ZPK_APROV = "N") e o vendedor solicita de novo, o MsSeek encontrará o registro e não vai criar novo — correto. Mas se ZPK_APROV = "S" e o vendedor solicita novamente, o sistema também não grava — pode ser intencional, mas não está documentado.
🟡 AVISO liberacaosolicitacao.tlpp / F44LPed
C5_TIPO hardcoded como "N"
Na montagem do ExecAuto, C5_TIPO está fixo como "N" (Normal). Se o pedido original tiver outro tipo (ex: "B" = bonificação, "L" = locação), o ExecAuto pode gerar erros ou alterar comportamento do pedido.
aAdd(aCabec, {"C5_TIPO", "N", Nil}) // ← sempre "N"!
aAdd(aCabec, {"C5_TIPO", SC5->C5_TIPO, Nil}) // ← pega do pedido
🟡 AVISO cadastro.tlpp
Nome do campo diverge da spec
A especificação define ZKC_CLIENTE mas o código usa ZKC_CLIENT (sem o E final) na chave primária do ModelDef. Precisa verificar se o SX3 foi criado com o nome correto.
// spec: ZKC_CLIENTE // código: ZKC_CLIENT ← 10 chars vs 9 chars
🔵 INFO M410STTS.PRW
Código de crédito misturado no PE
O ponto de entrada M410STTS contém lógica de controle de crédito (FFAT002, FFAT003, ZZA, FIE) completamente separada da lógica KG. É uma customização anterior reutilizando o mesmo PE. Cuidado ao modificar o arquivo para não quebrar o controle de crédito.
🔵 INFO tela.tlpp
Spec vs Código: campo ZKC_DESCRICAO
A especificação lista ZKC_DESCRICAO na tabela ZKC, mas o código não referencia esse campo em nenhuma parte. Ou o campo existe no SX3 e não é usado, ou foi esquecido.
